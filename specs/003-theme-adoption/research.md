# 研究记录：主题采用体验

## 决策：优先做 fork 后改造路径

**理由**：主题核心功能已经完成。下一阶段最能增加价值的是降低采用者第一次修改成本，让用户能快速把 Polyglow 改成自己的站点。

**放弃方案**：先做视觉变体。视觉变体能提升展示效果，但不能解决采用者“如何开始”的问题。

## 决策：提供最小内容模板

**理由**：当前示例内容适合验证多语言、分页和搜索，但对 fork 用户来说内容量偏大。最小模板能形成稳定起点。

**放弃方案**：只写文档让用户手动删除。该方式容易误删 About、Author 或必要图片，增加首次失败概率。

## 决策：增加 `theme:check`

**理由**：默认品牌、默认作者、默认域名和示例内容残留是主题采用中最常见问题。脚本检查能在发布前直接指出问题。

**放弃方案**：依赖人工检查。人工检查成本低，但容易遗漏。

## 决策：`theme:reset` 默认保守

**理由**：重置脚本不能意外删除用户内容。默认策略采用复制最小模板和输出确认信息；破坏性操作需要显式参数或人工确认。

**放弃方案**：默认清空内容目录。该方案速度快，但风险过高。

## 决策：beta 发布先使用 GitHub Release

**理由**：当前项目是 GitHub fork 型主题，GitHub Release 足够承载 beta 说明和采用入口。

**放弃方案**：立即做 npm create 模板。模板发布需要额外包名、版本策略和维护流程，不适合当前阶段。
