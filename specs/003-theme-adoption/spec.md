# 功能规格：主题采用体验

**功能分支**：`003-theme-adoption`

**创建日期**：2026-05-14

**状态**：草案

**输入**：用户描述：“Spec Kit 驱动，完善下一阶段开发计划；根据项目目标和开发进度，聚焦 fork 后低成本采用、最小内容模板、theme:reset、theme:check、beta 发布准备。”

## 用户场景与测试

### 用户故事 1 - 十分钟完成主题改造（优先级：P1）

开源采用者 fork 仓库后，能够根据一份短指南完成站点名称、域名、作者、About、示例内容和首页基础配置，得到一个能本地运行的个人内容站。

**优先级理由**：Polyglow 已具备发布能力，下一阶段核心价值是降低采用成本。采用者首次改造是否顺畅，决定主题能否被真实使用。

**独立测试**：从干净仓库开始，按照 fork 指南完成最小配置，运行本地开发命令，确认首页、文章页、About 和 Author 不再显示默认身份信息。

**验收场景**：

1. **假如** 采用者刚 fork 仓库，**当** 阅读 fork 指南，**那么** 能看到必须修改文件、可选修改文件和验证命令。
2. **假如** 采用者按指南完成最小配置，**当** 运行主题检查，**那么** 默认品牌、默认作者和示例内容残留会被报告。
3. **假如** 采用者完成配置并启动本地站点，**当** 访问 `/en/`，**那么** 页面展示采用者自己的站点信息和最小内容。

---

### 用户故事 2 - 一键获得最小内容模板（优先级：P2）

采用者能够保留主题结构，同时快速替换示例文章，获得一套最小可发布内容：英文文章、About、作者资料和必要图片。

**优先级理由**：当前示例内容用于验证多语言和分页，但对 fork 用户来说内容量偏多。最小模板能减少首次改造压力。

**独立测试**：运行内容重置流程后，确认示例文章被替换为最小内容，站点仍能构建并生成搜索索引。

**验收场景**：

1. **假如** 采用者决定不保留示例文章，**当** 使用重置流程，**那么** 仓库保留最小内容集和内容写作模板。
2. **假如** 内容重置完成，**当** 运行构建，**那么** 首页、文章页、About、Author、RSS、站点地图和搜索仍然生成。

---

### 用户故事 3 - 发布 beta 版本（优先级：P3）

维护者能够根据发布清单创建 `v1.0.0-beta.1`，并在 GitHub Release 中说明主题定位、使用方式、验证结果和已知非阻塞提示。

**优先级理由**：beta 发布能让主题进入可试用状态，同时保留后续根据反馈调整的空间。

**独立测试**：发布前运行检查命令，确认版本说明完整，tag 与 changelog 条目一致。

**验收场景**：

1. **假如** 维护者准备发布 beta，**当** 运行发布检查，**那么** 版本号、变更记录、发布说明和质量记录保持一致。
2. **假如** 发布 tag 已创建，**当** 查看 GitHub Release，**那么** 采用者能看到安装、配置、验证和已知提示。

---

### 边界情况

- 采用者只保留英文内容，但仓库仍配置十语言路由。
- 采用者没有 Cloudflare、R2 或统计服务凭据。
- 采用者误删 About、Author 或默认文章。
- 采用者忘记修改生产域名、作者名称或仓库链接。
- 发布前发现上游 Node/Astro 提示，但 Astro 诊断结果仍为 0 warnings。

## 需求

### 功能需求

- **FR-001**：项目必须提供 fork 后十分钟改造指南，列出必须修改文件、推荐修改文件和验证命令。
- **FR-002**：项目必须提供最小内容模板，覆盖文章、About、作者资料和图片引用。
- **FR-003**：项目必须提供内容重置流程，帮助采用者替换示例内容并保留主题结构。
- **FR-004**：项目必须提供主题检查流程，发现默认站点名、默认域名、默认作者、示例内容和占位链接残留。
- **FR-005**：发布前必须有 beta 发布清单，覆盖 tag、Release notes、CHANGELOG、验证命令和线上检查。
- **FR-006**：新增流程不得要求 R2、统计、CMS、评论、登录或远程对象存储。
- **FR-007**：新增流程必须保留本地图片和 `public/` 图片路径。
- **FR-008**：新增文档必须进入 README 和中文说明入口。
- **FR-009**：新增脚本必须有项目卫生测试或聚焦测试保护。

### 范围边界

- **范围内**：fork 指南、最小内容模板、内容重置流程、主题检查流程、beta 发布说明和相关测试。
- **范围外**：CMS、评论、登录、会员、支付、浏览器上传、强制 R2、npm create 模板和大型配置界面。
- **语言影响**：默认保留十语言配置；最小模板以英文为主，中文作为可选扩展示例。
- **内容影响**：涉及 posts、pages、authors、分类、标签、hero 图片和默认站点信息。
- **集成影响**：没有必需第三方服务。Cloudflare、R2 和统计继续保持可选。

### 关键实体

- **采用指南**：面向 fork 用户的最短改造路径，包含文件清单和验证命令。
- **最小内容模板**：能让主题独立运行的基础内容集合。
- **内容重置流程**：替换示例内容并保留主题结构的开发者流程。
- **主题检查流程**：发现默认值残留和配置缺口的发布前检查。
- **beta 发布包**：tag、发布说明、变更记录和验证结果的组合。

## 成功标准

### 可度量结果

- **SC-001**：采用者根据文档在 10 分钟内完成最小站点配置并启动本地预览。
- **SC-002**：主题检查能发现 100% 已定义的默认身份信息残留项。
- **SC-003**：内容重置后，`pnpm test`、`pnpm build` 和 Pagefind 索引生成成功。
- **SC-004**：beta 发布说明覆盖安装、配置、验证、部署和已知非阻塞提示。
- **SC-005**：README 和中文说明都能从首页入口进入采用指南、最小内容、主题检查和发布清单。

## 假设

- 下一阶段继续保持 Astro 原生和静态主题定位。
- 采用者主要通过 GitHub fork 使用主题。
- 采用者初次使用时优先需要英文最小内容。
- R2、统计和 Cloudflare 部署仍然作为可选集成处理。
- beta 发布先使用 GitHub Release，不新增 npm 发布流程。
